**作成日:** 2026-07-04
**ステータス:** 設計案 (実装はまだ)
**関連文書:** [alma_integration.md](alma_integration.md), [ldrp_extensions.md](ldrp_extensions.md), [future_work.md](future_work.md)

## 1. 背景・動機

### アイデア

タスク割当エージェント (現 PPO / 将来 ALMA Allocator) が、**タスクの配送状況に応じて稼働エージェント数を増減できる**仕様にする。割当方策の行動空間を `{タスク j, idle}` から `{タスク j, idle, off(退避) / on(投入)}` へ拡張し、fleet サイズ制御を「割当の一種」として同じ方策で扱う。

### なぜ LDRP で意味があるか (トレードオフ)

LDRP ではエージェントが増えるほど衝突・待機 (輻輳) が増え、**1 台あたりの移動効率は下がる**。一方でタスクを並列実行できる数は増える。つまり:

- エージェント数 ↑ → 移動効率 ↓ (輻輳デメリット) / タスク処理容量 ↑
- エージェント数 ↓ → 移動効率 ↑ / 処理容量 ↓

→ **タスク到着率・分布ごとに最適な稼働台数が存在する**はずで、これを配送状況から動的に学習・制御できれば、単なる省コストではなくスループット自体の改善が狙える (動的フリートサイジング × MARL)。

### 前提となる既知の事実 (2026-07 時点の調査結果)

| 部品 | 状態 |
|---|---|
| MAT-Dec actor の N 汎化 | per-agent 共有 MLP で重みは n_agents 非依存。**`obs_agent_id=False` で学習すれば異なるエージェント数にそのまま適用可** |
| MAT-Dec critic | 入力次元が `n_actions × n_agents` 連結で N 固定。ただし学習専用 (推論は actor 単独) |
| 推論経路 (test.py → PolicyRunner) | エージェントごとに `get_action(ag_idx, ...)` を呼ぶ構造なので、**呼ぶ回数を変えるだけで途中増減に自然対応**。ただし現 [PolicyRunner](../src/all_policy/policy_runner.py) は RNNAgent 専用 (`fc1.weight` を要求) で MAT-Dec モデルを読めない |
| drp_env | `agent_num` が `__init__` で固定。全配列 (goal_array / assigned_tasks / obs) が agent_num 長。スポーン/離脱 API なし |
| epymarl 学習基盤 | `EpisodeBatch` が `(batch, T, n_agents, …)` の固定形状。可変 N は非対応 |

## 2. 段階戦略

学習基盤まで触るとコストが大きいので 2 段階に分ける。**まず Phase A (方策実行時のみ変更可能な方式) から着手する。**

| Phase | 内容 | 増減の決定主体 | 学習基盤の変更 |
|---|---|---|---|
| **A. 実行時のみ** (先行) | 評価時にエージェント数を途中で増減できる。増減判断は固定ルール or 外部指示 | ルール (学習しない) | **不要** |
| **B. 増減も学習** (本命) | 割当エージェントが off/on を行動として学習 | 割当方策 | **必要** (下記 §5) |

Phase A は「env 側の増減基盤」の実装と検証を先に済ませる位置付けで、Phase B の土台になる。

---

## 3. 共通基盤: N_max + アクティブフラグ方式

可変長の配列伸縮 (動的スポーン) は env 全体・LaRe フック・SafeEnv まで波及して改修が大きい。代わりに:

1. **env は最初から `N_max` 体で生成**する (配列形状は全て N_max 固定)
2. 各エージェントに `active[i]` (bool) フラグを持たせる
3. 増減 = フラグの ON/OFF 切替として表現する

非アクティブ機の扱い:

- **観測**: obs をゼロ埋め (または退避状態を示す固定値)
- **行動**: `avail_actions` を「待機のみ」に制限 (epymarl の avail_actions 機構がそのまま使える。MAT-Dec は `logit[avail==0] = -1e10` でマスク済み)
- **衝突判定**: 除外 (退避ノードに居る想定。物理的にマップ上のノードを占有すると邪魔になるため、専用退避ノード or マップ外扱いにする)
- **報酬**: 0 (または稼働コスト 0)
- **タスク割当**: 割当対象から除外

利点: EpisodeBatch も critic も固定形状のままで済み、**epymarl 側はほぼ無改修**。SMAC の「死亡ユニット」の扱いと同じパターン。

### 設計判断が必要な 3 点

1. **退避中の機体の居場所**: 専用退避ノード (デポ) を map に追加するか、マップ外扱い (座標を無効値) にするか
2. **離脱時のタスク解放**: 離脱する機体が配達中タスクを持つ場合の扱い。ピック前なら `current_tasklist` に戻す ([alma_integration.md §3.4](alma_integration.md) の再割当と同じ解放処理 `assigned_list[old_idx] = -1` が流用可能)。**ピック後 (積荷あり) は離脱禁止にするのが簡明** (配達完了まで off を無効化)
3. **参入/離脱の遷移セマンティクス**: 即時テレポートか、「デポへ帰投 → 到着後に非アクティブ化」か。簡易版 (即時退避) から始める

## 4. Phase A: 方策実行時のみ変更可能な方式 (先行実装)

学習は一切変えず、**評価 (test.py / runner.py) でのみ**エージェント数を途中増減できるようにする。

### 4.1 必要な実装

| # | 対象 | 内容 | 規模 |
|---|---|---|---|
| 1 | [src/main/drp_env/drp_env.py](../src/main/drp_env/drp_env.py) | `active` フラグ + §3 の除外処理 (衝突・割当・報酬)。`set_agent_active(i, bool)` のような外部 API | 中 |
| 2 | [src/all_policy/policy_runner.py](../src/all_policy/policy_runner.py) | MAT-Dec ローダ追加: state_dict から `decoder.*` キーのみ抽出して `Decoder(input_shape, n_actions, n_embd)` に load する分岐 (critic の `encoder.*` は捨てる)。`obs_agent_id=False` 学習モデルが前提 | 小 (〜40 行) |
| 3 | [runner.py](../runner.py) | 評価ループで非アクティブ機をスキップ (`if not active[i]: continue`)。増減トリガの注入 (ステップ数指定 or 未割当タスク数の閾値ルール) | 小 |

### 4.2 増減トリガ (Phase A では学習しない)

固定ルールで動作検証する。例:

- 未割当タスク数 > 閾値 T_on で 1 台投入 / タスク枯渇が N step 続いたら 1 台退避
- または実験シナリオとして「step 200 で 2 台追加」のような外部スケジュール指定

これで「増減が正しく動く env」を学習なしで検証でき、かつ **固定 N との性能比較 (増減にそもそも価値があるか) を先に確認できる**。

### 4.3 前提条件

- 経路方策は `obs_agent_id=False` で学習した MAT-Dec (actor が N 非依存になる唯一の構成)
- 同一マップ (n_nodes が変わると obs 次元が変わり転移不可)
- 密度ロバスト性の注意: N 固定で学習した actor は特定の混雑度に過適合しうる。Phase A の評価で「5 体学習 → 7 体実行」の性能劣化度を測ること自体に意味がある

## 5. Phase B: 増減を割当エージェントが学習する方式 (本命)

### 5.1 行動空間の拡張

割当方策 (PPO / ALMA Allocator) の per-agent 行動を拡張:

```
b^a ∈ {タスク 0..|I|-1, idle, off}      # アクティブ機
b^a ∈ {on, stay-off}                     # 非アクティブ機
```

ALMA の pointer-net 構造なら「off」「on」を idle と同様の**学習可能な専用埋め込み**として追加するだけで済む ([alma_integration.md §6.1](alma_integration.md) の idle_embedding と同型)。

### 5.2 報酬設計 (最重要ハイパラ)

稼働コストを入れないと「常に全員 ON」に縮退し、強すぎると全員 OFF になる:

```
R = (タスク完了報酬) − c_active × (アクティブ台数) × Δt
```

- `c_active` (per-step 稼働コスト) が本質的なチューニング対象
- **対照実験**: コスト項なし (輻輳ペナルティ = collision/wait だけで最適台数に自然に落ちるか) vs コスト項あり。前者で成立するなら報酬設計がシンプルになる

### 5.3 学習基盤の変更点 (ここが大きい)

Phase B が「学習時にも大きな変更が必要」となる理由と、N_max 方式による回避策:

| 問題 | N_max + アクティブフラグでの解決 |
|---|---|
| EpisodeBatch の固定形状 | `n_agents = N_max` で確保すれば形状は不変。**epymarl のテンソル配管は無改修** |
| MAT-Dec critic の入力次元 | N_max 固定なので形状問題なし。ただし非アクティブ機の情報が critic 入力に混ざるため、ゼロ埋め or アクティブマスクを attention に渡す改修が望ましい |
| 非アクティブ機の学習信号 | 「待機のみ」の avail_actions マスクで行動は縛れるが、**loss 側でも非アクティブ機のエントロピー/方策勾配をマスクする**必要あり (mat_learner の mask 拡張) |
| 割当方策の遅延信用割当 | 台数変更の効果は数十〜数百 step 後に出る。**LaRe-Task の proxy 報酬と相性が良い** (割当決定の品質を直接反映)。ALMA の N_t 区間累積報酬でも可 |
| actor の密度過適合 | **学習中に初期アクティブ台数をランダム化 (2〜N_max)** して密度に汎化させる。既存のカリキュラム学習 (CL) の続きとして位置付く |

### 5.4 振動 (thrashing) 対策

ON/OFF が毎ステップ揺れる問題。[alma_integration.md §3.4](alma_integration.md) の再割当振動と同じ構造なので同じ緩和策を流用:

1. 切替コストを報酬に反映 (帰投・投入の無駄走行が学習信号に入るようにする)
2. ヒステリシス / 最小滞留時間 (off にしたら最低 K step は on にできない)
3. イベント駆動の粒度制限 (毎 step でなくイベント時のみ増減判断)

## 6. 評価設計

### 指標

- タスク完了数 (従来指標)
- **agent-step あたり完了数** (= 効率。増減制御の主張はこれで立てる)
- 平均稼働台数
- タスク待機時間 (作成 → ピックまで)
- 増減回数 / 振動率

### ベースライン比較

| 条件 | 内容 |
|---|---|
| 固定 N_max | 全台数常時稼働 (現状) |
| 固定 N (少数) | N = 2, 3, ... の各固定値 |
| ルール増減 (Phase A) | 閾値ルール |
| **学習増減 (Phase B)** | 割当方策が off/on を学習 |

**主張の形**: 学習増減が「固定のどの N よりもパレート上で良い」(完了数 − 稼働コストの両立) ことを示せれば強い。タスク到着率を変えたシナリオ (疎/密/バースト) で優位性が出るかも見る。

## 7. リスク・未決事項

| 項目 | 内容 | 緩和策 |
|---|---|---|
| 報酬バランス | c_active の設定次第で全 ON / 全 OFF に縮退 | §5.2 の対照実験で感度を先に測る |
| 退避ノードの物理設計 | マップにデポを足すと従来マップと非互換 | マップ外扱い (座標無効値 + 衝突除外) から始める |
| 積荷持ち離脱 | 配達中タスクの扱いが複雑 | ピック後は off 禁止で回避 (§3) |
| critic のマスク精度 | 非アクティブ機のゼロ埋めが value 推定を歪める可能性 | attention にアクティブマスクを渡す改修を Phase B 内で検討 |
| baseline 不変条件 | 既存挙動 (全機アクティブ固定) を壊さない | `use_dynamic_agents=False` (デフォルト) で従来と完全一致。CLAUDE.md の不変条件に整合 |

## 8. 実装順序

```text
[1] env: N_max + アクティブフラグ + 除外処理 (§3)      ← Phase A/B 共通基盤
[2] PolicyRunner: MAT-Dec decoder ローダ (§4.1-2)
[3] runner.py: 評価ループの増減対応 + 固定ルールトリガ (§4.1-3)
[4] Phase A 評価: ルール増減 vs 固定 N                  ← ここで価値を確認してから B へ
[5] 割当方策の行動空間拡張 + 報酬設計 (§5.1-5.2)
[6] 学習基盤のマスク拡張 + 初期台数ランダム化 (§5.3)
[7] Phase B 評価 (§6)
```

## 9. 関連環境・先行事例 (参考)

### 9.1 エピソード途中でエージェント数が変わる環境 (構造が近い)

| 環境 | 増減の仕組み | LDRP が借りられるもの |
|---|---|---|
| **Flatland** (NeurIPS 列車スケジューリング, [arXiv:2012.05893](https://arxiv.org/abs/2012.05893)) | 列車が出発時刻に**参入**し目的地到着で**退出**。グリッド上の衝突・デッドロックあり | **構造的に最も近い** (グラフ上 + 輻輳 + 参入/退出スケジュール)。参入トリガ設計 (出発時刻 = 外部スケジュール) は Phase A のルールトリガそのもの |
| **SMAC / SMACv2** | ユニットが戦闘で**死亡** = 途中減少。死亡ユニットは obs ゼロ埋め + `avail_actions` を no-op のみに制限 | **§3 の N_max + アクティブマスクの実装パターンそのもの**。epymarl は SMAC 用に作られており、**avail_actions 経由の無効化の配管は現学習基盤に既に通っている** → 実装の写経元 |
| **MAgent2** | 大規模戦闘で個体が死亡・population 動的変化 | 可変 N での観測設計 (局所観測 + 固定次元) |
| **Neural MMO** | エージェントが spawn/死亡するオープンエンド環境 | 参入 (spawn) 側のセマンティクス |
| **Multi-Agent Craftax** ([arXiv:2511.04904](https://arxiv.org/pdf/2511.04904)) | オープンエンド・大規模の動的 population | 最近のベンチマーク動向 |

### 9.2 参入/退出への適応を研究対象にした系譜 (Open MAS / Ad Hoc Teamwork)

- **GPL: Open Ad Hoc Teamwork Using Graph-based Policy Learning** (ICML 2021): チームメイトが途中で入れ替わる設定を graph NN で扱う。**可変数エージェントのネットワーク表現** (集合入力・GNN) の代表例。LBF / Wolfpack を open 化した改造例としても参考になる。MAT-Dec actor は per-agent MLP なので当面不要だが、critic を賢くする場合の参考
- **Transferable MARL with Dynamic Participating Agents** ([arXiv:2208.02424](https://arxiv.org/pdf/2208.02424)): 学習途中でエージェント数を変える転移の研究。§5.3 の「初期台数ランダム化」の裏付けに近い
- **オープンエンド協調の定式化 (DOSBG 系)** ([AAMAS 2025](https://link.springer.com/article/10.1007/s10458-025-09723-7)): 参加/離脱 + タスクストリームの定式化。論文化する際の問題設定の参考

### 9.3 フリートサイジングをドメインとして扱う系譜 (報酬設計の参考)

- **ライドヘイリング fleet management** (Lin et al., KDD 2018, [arXiv:1802.06444](https://arxiv.org/abs/1802.06444)): 車両の再配置・稼働を MARL で制御。**「稼働台数 × コスト vs サービス充足」の報酬設計** (§5.2 の `c_active`) の先行例
- **MAPD サーベイ** ([arXiv:2203.07092](https://arxiv.org/abs/2203.07092)): LDRP と同一のタスク定式化。MAPD 文献では必要台数は**オフライン事前決定が通例**で、「割当方策がオンラインで台数を決める」のはギャップ = **本設計の新規性の主張点**に使える
- **Scalable MARL for Warehouse Logistics** ([arXiv:2212.11498](https://arxiv.org/pdf/2212.11498)): 倉庫でロボット + 人間ピッカーの大規模割当。中央割当 + 移動エージェントの階層構造が ALMA 設計と近い

### 使い分けの目安

- **実装の写経元**: SMAC の死亡マスキング (epymarl 内の既存配管) → §3
- **参入/退出セマンティクス**: Flatland (スケジュール参入・到着退出) → §4.2
- **可変 N のネットワーク表現**: GPL → Phase B の critic 改修時
- **報酬・評価設計**: ライドヘイリング系 (稼働コスト) + MAPD サーベイ (新規性の位置づけ) → §5.2 / §6
